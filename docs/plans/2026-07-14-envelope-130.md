# Soporte del formato AEAT "con sobre" (`<T…>`) — Modelo 130

> **Para Claude Code:** implementa esto con `superpowers:executing-plans`, tarea a tarea.
> Antes de empezar, **confirma con el usuario el alcance (opción A o B)** — el plan deja
> ambas abiertas. Mantén `mod130.boe` (plano) y los tests del 303 verdes (backward-compat).

**Rama:** `feat/envelope-130` (creada off `main`). Para la **opción B** conviene mergear/rebasar
`feat/modelo303` primero (aporta el multi-`record` sobre el que se construye el envelope).

## Objetivo

Que `boeidl` lea y escriba el **fichero que la AEAT realmente acepta** (envuelto en tags
`<T130…>` / `<AUX>` / `</T130…>`), no solo el diseño de registro plano. Primer entregable: el
**Modelo 130 con sobre**, byte-idéntico a los ficheros que se presentan/importan en la Sede.

## Contexto del repo

- `boeidl` compila un `.boe` (diseño de registro AEAT) → Rust que serializa/deserializa/valida.
- `main`: `models/mod130.boe` modela el **DR130e15v11 plano** (registro único de 878 chars, sin
  sobre, importes de 13).
- `feat/modelo303`: añadió **múltiples `record` por fichero**, pero dejó **fuera de scope el
  sobre y la composición de páginas** (a Rust a mano) — ver `docs/plans/2026-04-08-modelo303.md`.

## El problema: hay DOS familias de formato

| | **DR plano** (lo que boeidl modela hoy) | **Con sobre `<T…>`** (lo que se presenta de verdad) |
|---|---|---|
| Envoltorio | ninguno (registro único) | `<T130 0 EEEE PP 0000><AUX>…</AUX>` + página + `</T1300 EEEE PP 0000>` |
| Ancho importes | 13 (signo+12) | **17** (`N`+16), importe×100, sin coma |
| Posición [01] | 77 (del registro) | 109 (de la página) |
| ¿Fichero que acepta la Sede? | (canal lote, sin confirmar) | **Sí — validado byte-exact contra justificante real Y en vivo importando en `PAMW-M130/E2026`** |

El fichero real presentado del 130 empieza por `<T130020262T0000><AUX>…` y mide **946 bytes**
(sobre 328 + página 600 + cierre 18).

## Fuente de verdad ya validada (reutilízala)

En `~/Dev/aldoborrero/autonomo/apps/sheets/` hay una implementación en TypeScript **byte-exact
validada** contra ficheros reales de la AEAT:
- `src/lib/modelo130.ts` — mapa de casillas del 130 con sobre (posiciones/anchos exactos,
  encoding, `an`/`num`/`numN`). **Spec autoritativo del layout con sobre.**
- `src/lib/modelo303.ts` + `src/lib/spec303.json` — el 303 con sobre (multi-página), para generalizar.
- `test/modelo130.test.ts` — cifras reales.

Si tienes acceso a ese repo, léelo. Si no, el layout está embebido abajo.

### Layout del 130 con sobre (946 bytes)

**SOBRE (328 bytes):**
```
1-2 "<T"  3-5 "130"  6 "0"  7-10 ejercicio(AAAA)  11-12 periodo(1T..4T)  13-17 "0000>"
18-22 "<AUX>"  23-92 blancos(70)  93-96 versión programa  97-100 blancos(4)
101-109 NIF entidad desarrolladora(9, blanco si no aplica)  110-322 blancos(213)  323-328 "</AUX>"
```
**PÁGINA (600 bytes, registro DR 13001):**
```
1-11 "<T130" "01" "000>"   12 indicador compl.(espacio)   13 tipo declaración (B|G|I|N|U)
14-22 NIF(9)  23-82 apellidos(60)  83-102 nombre(20)  103-106 ejercicio  107-108 periodo
[01] 109-125  [02] 126-142  [03] 143-159(signo)  [04] 160-176  [05] 177-193  [06] 194-210  [07] 211-227(signo)
[08] 228-244  [09] 245-261  [10] 262-278  [11] 279-295(signo)
[12] 296-312  [13] 313-329  [14] 330-346(signo)  [15] 347-363  [16] 364-380  [17] 381-397(signo)  [18] 398-414  [19] 415-431(signo)
432 marca complementaria("X"/espacio)  433-445 justificante anterior(13)  446-479 IBAN(34)
480-575 reservado(96, blancos)  576-588 sello electrónico(13, blancos)  589-600 "</T13001000>"
```
Cada importe: 17 chars, `importe×100` alineado a la derecha con ceros; los `(signo)` llevan `N`
en la 1ª posición si son negativos (si no, dígitos).

**CIERRE (18 bytes):** `"</T1300"` + ejercicio(4) + periodo(2) + `"0000>"`.

**Derivaciones (para los `check`):** `[03]=[01]-[02]` · `[04]=20%·max([03],0)` ·
`[07]=[04]-[05]-[06]` · `[12]=[07]+[11]` · `[14]=[12]-[13]` · `[17]=[14]-[15]-[16]` · `[19]=[17]-[18]`.

**Encoding:** ISO-8859-1, mayúsculas, sin tildes pero **conservando Ñ y Ç** — que es justo lo que
`crates/boeidl/src/runtime/sanitize.rs` ya hace bien. (El TS lo hace en US-ASCII, que machaca
Ñ→N; **NO copies eso**.)

## Dos caminos — decide con el usuario

### Opción A — MVP pragmático (recomendada si el foco es solo el 130)
Un `.boe` que describa **la página** (600 chars, anchos de 17, en sus posiciones) + **Rust a mano**
que envuelva el sobre `<T130…>`/`<AUX>`/cierre. Reutiliza la arquitectura que el 303 ya adoptó.
Desbloquea el 130 con sobre rápido.

### Opción B — extensión del DSL (envelope nativo)
Que el `.boe` exprese el sobre entero. Opt-in y sin romper `mod130.boe` plano:
1. **`grammar.pest`**: bloque `envelope { header = <tmpl>  trailer = <tmpl>  contains = [seg, seg?, …] }`;
   regla de **plantilla** (literal + `${ident}`); marca `when <bool_expr>` y `repeat` para segmentos.
2. **`ast.rs`**: `Envelope`, `Segment { record, wrap: Option<(Template,Template)>, when, repeat }`,
   `Template = Vec<Part(Lit|FieldRef)>`; campos compartidos (ejercicio/periodo/model) referenciables
   desde header/trailer/páginas.
3. **`parser.rs`**: parsear lo nuevo.
4. **`validator.rs`**: existencia de `${campo}`; coherencia ejercicio/periodo header↔trailer; suma
   de longitudes de segmentos; `when` válido.
5. **`codegen/rust.rs`**: struct top-level `Mod130File` que compone segmentos; `marshal` (header
   interpolado + segmentos con wrap + condicionales + trailer); `unmarshal` (verifica tags fijos en
   offset, extrae campos del header, parsea cada segmento).
6. **`runtime/`**: helper para verificar delimitadores. Reusa `encode`/`sanitize`.

Ventaja: el sobre es **todo longitud fija**, así que el `unmarshal` sigue siendo offset-based con
los tags como literales — **no hace falta parser dinámico**.

### Recomendación
Solo 130 → **A** (rápido, reusa nuestro mapa validado). La **B** solo amortiza al ir a por
303/349/390 con sobre — hazla **entonces**, con el aprendizaje real de haber hecho el 130 a mano,
para que el diseño del DSL salga de un caso concreto y no de especular.

## Criterio de "hecho"

- **Golden byte-exact**: generar un 130 de ejemplo y que el fichero salga idéntico a un `.txt`
  presentado real (946 bytes, empieza `<T130…`). Pide cifras/fichero real al usuario.
- **Round-trip**: parsear un `.txt` real → struct → re-serializar → mismos bytes.
- `mod130.boe` plano y los tests del 303 **siguen verdes**.
- Los `check` (derivaciones) validan contra las cifras reales.
